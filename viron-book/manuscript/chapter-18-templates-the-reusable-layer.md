# Chapter 18 — Templates: The Reusable Layer

*Every time you open a blank note and start a new project, you make small decisions.*

What sections should this contain?
Should I define the outcome upfront?
Where do decisions go?
What questions should I answer before I start work?

These decisions take seconds each.

Over a year of knowledge work, they add up to hours.

Worse, they produce inconsistency.

A project note from January has different sections than one from March.
The March note has a section the January note lacks.
During Weekly Review, you spend time reorienting to each note's idiosyncratic structure before you can do the governance work.

Templates eliminate this.

Not because they make every note look the same.
Because they externalize the decisions that do not need to be made each time.

## What Templates Are Not

Templates are not forms.

A form is rigid.
You fill in the blanks or you have failed to complete the form.

A template is structural memory.
It presents the questions you want to answer.
It does not require every field to be used every time.

A project that has no open questions should not have a fabricated open question to fill the field.

Leave the field blank.
Or delete it for that note.

The template is a starting point.
Not a cage.

## Why Consistency Matters

Consistent structure serves review more than it serves creation.

When you create a note, you know what you are writing.
Context is fresh.
You can navigate an idiosyncratic structure because you built it.

Six weeks later, during review, that context is gone.

You open the note.
You need to quickly answer: what is the outcome, what is the next action, what decisions have been made?

If those questions live in predictable locations, review takes thirty seconds per note.

If each note is structured differently, review requires re-reading the full document before governance can begin.

> Consistency is an investment in your future self.

## The Templates Folder

Templates live inside the VIRON folder in Apple Notes.

The folder name is: Templates

This folder contains exactly four notes:

- Project Note
- Meeting Note
- Weekly Review
- Library Note

Do not add additional templates unless a recurring note type clearly justifies one.

Each additional template is a maintenance obligation.
Templates that are rarely used decay through disuse.
Decayed templates create confusion.

Four templates.
Four recurring patterns.

## Template 1 — Project Note

### When to Use It

Create a project note whenever work requires:

- More than one task to complete
- More than one week to finish
- Coordination of decisions, research, or context

If the work requires only one task and can be completed this week, it is a task in Reminders, not a project.

### The Full Template

```
Project: [Name]
Outcome: [One sentence — what does done look like?]
Start Date:
Target Date:

---

## Scope

What is included:

What is excluded:

---

## Milestones

- [ ]
- [ ]
- [ ]

---

## Decisions

[Significant choices made and why. Date each entry.]

---

## Open Questions

[Unresolved items that need answers before proceeding.]

---

## Notes / Context

[Background, research, links, evolving thinking.]
```

### Field by Field

**Outcome**

The most important field.

This is the finish line.
One sentence.
It describes what will be true when the project is complete — not what you will do, but what will have changed.

Weak: "Work on the website redesign."
Strong: "The website redesign is complete and live, with updated copy on all five main pages."

If you cannot write this sentence, the project is not ready to begin.

**Scope**

Two parts: what is included and what is explicitly excluded.

The exclusion list is often more valuable than the inclusion list.

It prevents scope creep before it starts.
When someone asks "could we also add X?" you check the exclusion list.
If X is on it, the answer is no — not because you are inflexible, but because you made a deliberate decision when scope was clear.

**Milestones**

Observable checkpoints.

Not tasks.
Not progress notes.

A milestone is something you can point to and say: "This happened."

"Draft complete" is a milestone.
"Working on draft" is not.

Keep milestones to three to five.
More than five and the project needs to be split.

**Decisions**

A dated log of significant choices made.

Not every decision.
Decisions that someone could second-guess later, or that required genuine tradeoffs.

"March 4: decided to delay the integration to phase 2. Reason: insufficient API documentation from vendor."

This field prevents re-litigation.
It also creates a record of your reasoning that becomes useful Library material after the project closes.

**Open Questions**

The questions you cannot yet answer.

Writing them explicitly externalizes the anxiety they create.

An open question held in memory creates background tension.
An open question written in a note creates a visible placeholder.

Review this field during Weekly Review.
Questions that have been answered get removed.
Questions that have persisted across multiple weeks may be blockers that need escalation.

## Template 2 — Meeting Note

### When to Use It

Every meeting with an external party.
Every meeting where decisions are made.
Any recurring meeting that produces commitments.

Do not skip the meeting note for "quick calls."
Quick calls produce decisions.
Undocumented decisions become disagreements.

### The Full Template

```
Meeting: [Name / Participants]
Date:
Project:

---

## Objective

[What is this meeting trying to resolve or advance?]

---

## Key Points

-
-
-

---

## Decisions Made

-
-

---

## Action Items

- [ ] [Task] — [Owner] `#task`

---

## Open Questions

[What remained unresolved?]
```

### Field by Field

**Objective**

Write this before the meeting begins, not after.

It defines what success looks like for the meeting.

"Agree on the pricing structure" is an objective.
"Discuss pricing" is an agenda item, not an objective.

The objective prevents meetings from wandering.
When conversation drifts, the objective provides the redirect.

**Key Points**

Substantive observations, context, and information shared.

Not a transcript.
The highlights.

These stay in the note after action items are moved to Reminders.

**Decisions Made**

Record every commitment made in the meeting that constitutes a decision.

Separate from action items.

A decision is an agreement about direction.
An action item is a task that implements the decision.

"We agreed to use the external vendor rather than building in-house" is a decision.
"Research vendor pricing by Thursday" is the action item that follows from it.

Both get recorded.
Only the action item moves to Reminders.

**Action Items**

Format every action item as: task — owner — tag.

After the meeting, highlight each action item and send it to Reminders.

The task title becomes the Reminder title.
The note field in Reminders contains a backlink to the meeting note.

Then delete or check off the items from the meeting note.
The meeting note should reflect context and decisions.
Reminders holds execution.

## Template 3 — Weekly Review

### When to Use It

Every week.
Same time.
Same structure.

### The Full Template

```
Weekly Review — [Date]

---

PROJECTS

[ ] Review each active project
    - Outcome still valid?
    - Next action explicit?
    - Scope unchanged?

Notes:


---

INBOX

[ ] Process all Inbox items to zero

Items processed: __
Items deleted: __
Items moved to Library: __
Items moved to Projects: __

Notes:


---

REMINDERS

[ ] Audit VIRON Tasks
[ ] Audit each project list
[ ] Audit Flagged and Scheduled
[ ] Process Reminders Inbox

Notes:


---

CALENDAR PREVIEW

[ ] Review next 7 days
[ ] Create deep work blocks where needed
[ ] Remove or decline low-value meetings

Key dates next week:


---

FOCUS THEME

Next week I am focused on:


---

CARRY-FORWARD NOTES

```

### Why the Template Has Checkboxes

Checkboxes create completion accountability.

Without them, review becomes scanning.
Each section is visually confirmed as complete.

The temptation to skip a section is reduced when skipping means leaving a visible unchecked box.

This sounds trivial.
In practice, it changes behavior.

### The Notes Field in Each Section

The notes field under each section is optional.

Use it when something needs to carry forward.

"Projects: noticed that Client A has not had a next action in three weeks. Needs decision by next Friday."

That note lives at the top of next week's review.

## Template 4 — Library Note

### When to Use It

When an idea has become a principle.

Not when an idea is interesting.
Not when an insight is recent.

When you can state the principle clearly, give it a title that would help you find it later, and connect it to at least one concrete example from your experience.

That standard filters out most Inbox notes.
It should.

The Library is not a collection.
It is a curation.

### The Full Template

```
[Title — a clear, searchable principle]

---

[Body — 50 to 200 words]
The principle stated plainly.
When it applies.
What it looks like in practice.
One concrete example from experience.

---

Source: [Project or context — optional]
Tags: #[thematic] #[thematic]
```

### The Title Standard

The title should be a complete sentence that captures the principle.

Not: "Pricing"
Not: "Client conversations"
Not: "What I learned about scope"

Useful: "Clients interpret silence as confirmation. Restate agreements explicitly."
Useful: "Volume discount conversations happen in the second client touchpoint, not the first."
Useful: "Scope defined without explicit exclusions will expand."

A title you cannot use in a search is a title that will never help you find the note.

Write it as a sentence.
Keep it specific.
Make it retrievable.

### Length Discipline

Fifty to two hundred words.

If the note requires more than two hundred words to state the principle, it has not been fully distilled.

Keep compressing until the core is clear.

A well-distilled Library note can be read in sixty seconds.
It delivers its insight on the first read.
There is no context to rebuild.

## When to Deviate from a Template

Templates are starting points.

Deviation is acceptable when:

- A project has no meaningful milestones (skip the field)
- A meeting produces no action items (leave the section empty)
- A Library note requires more than two hundred words to state a genuinely complex principle (use judgment)

Deviation is not acceptable when:

- Filling in the field feels inconvenient
- You are in a hurry and skipping seems harmless
- You are experimenting with a new structure mid-project

The second category is not deviation.
It is erosion.

Erosion compounds.
Each skipped field makes the next skip easier.
Within weeks, templates become suggestions.
Within months, they are ignored.

> Templates serve you during review, not during creation.
> The investment is made at creation.
> The return is earned during governance.

## Creating Templates in Apple Notes

Open Notes.
Navigate to VIRON → Templates.
Create a new note for each template.
Copy the template text into each note.

When starting a new project:

1. Open the Project Note template.
2. Select all text.
3. Copy.
4. Create a new note in the Projects folder.
5. Paste.
6. Fill in the header fields.

This process takes under sixty seconds.

If your workflow changes and a template field becomes irrelevant, update the template.

Do not maintain fields you never use.
A template that does not reflect actual practice is a template that creates friction rather than removing it.

Review and update your templates annually.
The goal is not permanence.
The goal is relevance.
